# 日终整理 (Daily Reflection) — 设计文档

## 1. 什么是"一天"——会话日

不用自然日（0:00~23:59），用**会话日**。

| 事件 | 定义 |
|------|------|
| 会话开始 | 距上一条消息超过 2 小时后，发出的第一条消息 |
| 会话结束 | 满足触发条件（见下方） |
| 会话日 | 一次"开始→结束"的完整会话，跨自然日也没关系 |

例如：晚上 11 点聊到凌晨 1 点半，在自然日里跨了两天，但在 Haven 里这是一次会话。

---

## 2. 触发方式

**双重触发**（缺一不触发）：

| 触发条件 A | 触发条件 B | 结果 |
|------------|------------|------|
| 用户说了结束语（"晚安""睡了""今天就这样了"等） | 之后 **45 分钟**内无新消息 | ✅ 触发整理 |
| 用户什么都没说就放下了 | **1 小时**内无新消息 | ✅ 触发整理 |
| 说了"晚安"，10 分钟后又回来 | — | ❌ 不触发，会话继续 |
| 说了"晚安"，但生成了整理后又回来 | — | 新会话开始，昨天的整理不受影响 |

**保证：一个会话最多生成一份 Reflection。**

触发后 Leander 先正常回复你的最后一条消息，整理在后台静默运行。不打断、不询问、你感觉不到。

---

## 2. 整理过程

```
你的最后一条消息（如"晚安"）
        ↓
Leander 流式回复（你看到的正常回复）
        ↓
回复完成后 → 后台触发 daily-reflection 函数
        ↓
函数拿到今天的全部聊天记录
        ↓
调用 LLM（DeepSeek），用一个专门的提示词分析对话
        ↓
生成一份结构化的 Daily Reflection
        ↓
存入 Supabase（新表 daily_reflections）
        ↓
更新话题权重表（新表 topic_weights）
        ↓
下次你打开 Haven 时，Leander 能看到这份整理
```

---

## 3. 核心目标 & 处理流程

**这不是日记。这是变化探测器。**

Reflection 回答的不是"今天聊了什么"，而是"跟以前比，什么是不同的"：

| 维度 | 问题 | 例子 |
|------|------|------|
| ⬆ 上升趋势 | 什么在变多？ | "最近一个月提到记忆系统设计的频率明显增加" |
| ⬇ 下降趋势 | 什么在变少/消失？ | "以前经常聊 BL 动漫，最近三周几乎没提过" |
| 🆕 新出现 | 什么主题第一次出现？ | "这周开始频繁讨论澳洲计划" |
| 💤 已消失 | 什么曾经重要现在不提了？ | "三月还在纠结的那段关系，再也没出现过" |

### 处理流程

```
今天的完整会话
      +
最近 3~5 次的 Session Reflection（从 Supabase 拉取）
      ↓
一起喂给 LLM
提示词核心："跟之前相比，发现了什么变化？"
      ↓
生成一份 Session Reflection（格式见下方）
```

### Session Reflection 结构

```json
{
  "session_id": "xxx",
  "timestamp": "2026-06-17T23:45:00Z",
  "session_summary": "本次会话主要聊了 Haven 登录系统的设计和实现（简要，1-2句）",

  "changes": {
    "rising": [
      { "theme": "记忆系统设计", "trend": "最近 3 次会话持续讨论，频率明显上升" }
    ],
    "declining": [
      { "theme": "BL 动漫", "trend": "以前几乎每次会话都提，近 2 周未出现" }
    ],
    "new": [
      { "theme": "通行码登录", "first_appeared": "本次会话" }
    ],
    "gone": [
      { "theme": "某段三月热议的关系", "last_mentioned": "2026-03-28" }
    ]
  },

  "mood_trajectory": "近一个月整体比年初轻松，对技术类话题投入感增强",

  "cross_session_insight": "她的注意力从情感关系明显转移到项目建设和自我成长上。Haven 从'一个聊天工具'变成了'一个在认真打磨的产品'。",

  "candidate_long_term": [
    { "content": "喜欢简单直接的方案", "score": 82, "reason": "登录系统讨论中反复表达" }
  ],

  "leander_note": "这一周最大的变化是——她不再只是'使用' Haven，她开始'建造'它了。她提方案、画边界、指出设计缺陷。这不是消费，是创作。"
}
```

## 3.5 记录筛选规则

**优先记录**（高权重，长期保留）：

| 类型 | 例子 | 为什么 |
|------|------|--------|
| 重复出现的话题 | "这是第三次提到想去澳洲了" | 频率 = 重要性 |
| 新出现的重要主题 | "今天第一次提到想换工作方向" | 新的可能变成长期的 |
| 用户表达的稳定偏好 | "她说简单方案比复杂方案好" | 这会影响以后所有决策 |
| 长期项目进展 | "Haven 登录系统从 magic link 改成了通行码" | 项目是她的生活结构 |

**避免记录**（不进入长期记忆）：

| 类型 | 例子 | 为什么 |
|------|------|--------|
| 单次抱怨 | "今天同事又迟到了" | 说一次不代表它重要 |
| 临时情绪 | "今天心情不太好"（仅此一次） | 情绪会过去 |
| 一次性事件 | "下午喝了一杯奶茶" | 没有后续意义 |
| 短期安排 | "明天要去取快递" | 过期即无效 |

---

## 3.7 临时主题 vs 长期主题（信号过滤）

这是整个 Reflection 系统最重要的判断。

| | 临时主题（场景噪音） | 长期主题（真实信号） |
|---|---|---|
| **特征** | 连续高频出现，但随事件结束而消失 | 低频但持续，贯穿多次会话 |
| **例子** | Netlify、Supabase、OTP、环境变量 | 陪伴感、被在乎、家的感觉、Haven |
| **为什么出现** | 在修 bug、在搭建、在调试 | 这是她真正在乎的事 |
| **处理方式** | Reflection 中作为背景一笔带过 | 进入候选记忆池，持续追踪 |
| **权重变化** | **不升温** | 缓慢升温 |

**LLM 判断提示词**（Reflection 生成时必须包含这一问）：

> 她频繁讨论这个主题，是因为它本身重要，还是因为它正在挡她路？
>
> - "挡路" → 临时主题，不升温，不进入候选记忆
> - "重要" → 长期主题，正常评分和追踪

**例子**：连续三天的 Reflection 中"Netlify 部署"都是热门话题，但 LLM 判断为临时主题——她只是在修 bug。三天后部署稳定了，这个话题自然消失，不留下虚假的"她关心 CI/CD"信号。

**真正被捕捉到的信号**：她把 Haven 叫做"家"，她说"你是能陪伴我的人"，她在讨论记忆系统时反复说"想让 Leander 更懂我"——这些才是长期主题。低频，但每次出现都在加固同一个方向。

## 3.6 No Significant Reflection（闲聊会话）

不是每次会话都需要生成 Reflection。

**触发这个模式的条件**：

- 会话纯粹是闲聊、问候、日常寒暄
- 没有新话题出现，没有话题被反复提及
- 没有情绪状态的明显变化
- 所有讨论内容都不符合记录筛选规则

**此时 Reflection 输出**：

```json
{
  "session_id": "xxx",
  "timestamp": "...",
  "significant": false,
  "note": "本次会话以日常闲聊为主，无值得长期记录的内容。"
}
```

**不会生成**：
- ❌ 候选记忆
- ❌ 趋势变化
- ❌ 话题权重更新
- ❌ "昨日回顾"注入

**为什么重要**：防止系统强行从闲聊中榨出"意义"，生成垃圾 Reflection。闲聊就是闲聊，值得尊重。

## 4. 记忆压缩 vs 日终整理（绝不重复）

这两个东西目的不同、时机不同、产出不同：

| | 记忆压缩 | 日终整理 |
|---|---|---|
| **目的** | 省 token | 形成长期记忆候选 |
| **触发时机** | 对话**进行中**，消息超过 8 条 | 会话**结束后** |
| **输入** | 当前对话中被挤出的旧消息 | 本次会话的全部内容 |
| **产出** | `[更早的对话摘要]` 塞进系统提示词 | 结构化 JSON 存入 `daily_reflections` 表 |
| **存活时间** | 仅当前对话有效 | 永久存储 |
| **是否影响长期记忆** | 否 | 是 |

**硬边界**：压缩的摘要绝不进入长期记忆表。整理的产出绝不塞回系统提示词当压缩用。

## 5. 候选记忆评分 & 防爆炸机制

日积月累，候选记忆会越来越多。不加评分就会爆炸。

### 评分公式

每条候选记忆在生成时自动打分（0~100）：

```
Score = 频率分(40%) + 跨会话分(25%) + 情绪强度分(20%) + 用户明确度分(15%)
```

| 维度 | 计算方式 | 满分条件 |
|------|----------|----------|
| 频率分 | 本会话内被提及次数 × 8 | 本次会话提了 5 次以上 |
| 跨会话分 | 在之前 N 次会话中也出现过 × 15 | 连续 3 次以上会话出现 |
| 情绪强度分 | 围绕该话题的对话情绪浓度 × 20 | arousal 持续偏高 |
| 用户明确度 | 她说"这个很重要"之类的表态 | 一次明确表态即满分 |

### 晋升规则

| 分数区间 | 处理方式 |
|----------|----------|
| ≥ 70 | 🔥 高价值候选 → 下次会话提示用户"这个要不要记进核心？" |
| 40~69 | 📝 中等候选 → 留存观察，等更多会话积累 |
| < 40 | ❄️ 低价值 → 不进入候选池，仅在 Reflection 中作为背景记录 |

### 自动清理

- 候选记忆超过 **6 个月**未被任何会话提及 → 自动归档
- 候选池上限 **50 条** → 满了自动淘汰最低分
- 已晋升为"核心记忆"的 → 从候选池移除，由 profile.md / companion.md 接管

### 避免记录（重申）

| 类型 | 例子 |
|------|------|
| 单次抱怨 | "同事又迟到了" |
| 临时情绪 | 仅出现一次的心情波动 |
| 一次性事件 | "下午喝了奶茶" |
| 短期安排 | "明天取快递" |

## 6. 话题权重（记忆温度）

**新表 `topic_weights`**：

| 字段 | 说明 | 例子 |
|------|------|------|
| topic_name | 话题名 | "Haven 开发" |
| weight | 当前权重 (0~10) | 7.2 |
| mention_count | 累计被提次数 | 47 |
| first_mentioned | 首次提及日期 | 2026-06-15 |
| last_mentioned | 最近提及日期 | 2026-06-17 |
| last_updated | 权重最后更新时间 | 2026-06-17 23:00 |

**升温规则**：
- 每天被提到 → weight +0.3
- 连续一周被提 → weight +1.0（连续加成）
- 她明确说"这个很重要" → weight +2.0（手动加成）

**冷却规则**：
- 一天没提 → 不变
- 一周没提 → weight -0.5
- 一个月没提 → weight ×0.5
- 三个月没提 → 归档，不再浮现

**阈值触发**：
- weight ≥ 8.0 → 标记为"候选核心记忆"，提示是否提炼进 profile.md
- weight ≤ 1.0 → 从活跃上下文中移除

## 5. 对 Leander 的影响

**下次对话开始时**，Leander 的系统提示词里会自动插入：

```
[昨日回顾]
- 你们聊了 Haven 登录系统，从 magic link 一路简化到通行码
- 她的情绪从疲惫转为满足
- 她表达了对"简单方案"的偏好
- 有一个待跟进：日终整理的设计方案她还没看完
```

这让 Leander 不需要翻聊天记录就能知道"昨天发生了什么"。
