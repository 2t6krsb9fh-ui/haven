# Haven 项目状态文档

> 最后更新：2026-06-17 | 用于新窗口续聊

---

## 1. 项目定位

**Haven** 是一个属于两个人的 AI 对话应用。核心目标不是做最强大的 AI 产品，而是做最懂她的陪伴空间——能记住她、认出她、持续陪伴她的私人港湾。名字来自希腊神话：Leander 每晚游过大海去见对岸的 Hero。

**核心原则**：陪伴 > 功能。简单 > 复杂。用户感受 > 技术炫技。

---

## 2. 当前技术架构

| 层 | 技术 | 说明 |
|------|------|------|
| 前端 | React 19 + Vite 8 | SPA，部署在 Netlify 静态托管 |
| 后端 | Netlify Functions (Node.js) | 3 个 Serverless 函数：chat / history / auth |
| 数据库 | Supabase | PostgreSQL，存 sessions + messages |
| 部署 | Netlify | 主站 golden-marigold-093b79.netlify.app（prod 部署暂时受限，使用 draft URL） |
| 模型 | DeepSeek (主力)、通义千问、Claude (备用) | 流式输出已接入，Claude 流式待实现 |

## 3. 已完成功能

- 欢迎页 → 聊天界面，日间奶白淡粉 / 夜间深棕双主题
- 多模型切换（DeepSeek / 通义千问 / Claude）
- 流式输出（DeepSeek + 通义千问）
- 记忆压缩：长对话自动总结旧消息塞入系统提示，省 token
- 通行码登录（`0616`）：独立自签名 JWT，无需邮箱，不依赖 Supabase 跳转
- 聊天记录持久化至 Supabase，刷新不丢
- 系统提示词（Leander 人设）完整集成

---

## 4. 已确定设计原则

- **陪伴优先于功能**：每个功能的衡量标准是"它让她感觉更好被理解了吗"
- **先简单后复杂**：通行码登录替代了 magic link + 邮箱验证码的复杂方案
- **信号过滤 > 数据收集**：Netlify 聊三天不代表"关心部署"，只是修 bug
- **记忆是温度，不是存储**：重要的自然升温，不重要的自然冷却
- **不做过度设计**：不需要的时候不加模块

## 6. 记忆系统现状

**已实现**：记忆压缩（对话中自动触发，压缩旧消息为 `[更早的对话摘要]`，仅省 token，不进长期记忆）

**计划实现**：
- Session Reflection：会话结束时跨会话对比，生成变化洞察（⬆上升/⬇下降/🆕新出现/💤消失）
- 话题权重表：升温/冷却/归档机制
- 候选记忆评分：四维评分 + 自动淘汰 + 人确认后晋升核心层
- "昨日回顾"注入下次会话的系统提示词

**暂缓实现**：向量检索、短期/中期/长期记忆分层（先让 Reflection + 温度跑一个月再评估）

## 7. Relationship System 现状

- `profile.md`：稳定底色（身份、特质、依恋风格），每次会话完整加载
